Looking Back

SPN Ciphers use a repeated application of XOR key mixing, substitution, and permutation to achieve confusion and diffusion. Feistel Ciphers break designing a good cipher down into key expansion/schedule algorithm and round function F. DES, 3DES, and AES are examples of block ciphers, which are symmetric key algorithms. Currently, only AES is recommended for use.

Evaluation Criteria

  1. Security: Resistance to cryptanalysis, soundness of math, randomness of output, etc
  2. Cost: Computational efficiency (speed) and memory requirements, etc
  3. Algorithm and Implementation Characteristics: Flexibility, hardware and software suitability, algorithm simplicity, etc

In that order!

AES Winner - Rijndael

  • Excellent performance across platforms
  • Good security margin
  • Well-suited to smart cards due to low memory (RAM and ROM) requirements
  • Operations defend well against attacks on smart card implementations
  • Fast key setup
  • Supports expanded key and block sizes
  • Good instruction-level parallelism

One Time Pads

If the key is truly random and is the same length as the plaintext, then the ciphertext is also truly random. What we have is a one-time pad. A one-time pad is a chunk of key material that is as long as the plaintext to be encrypted and that, once it is used, is thrown away and never used again for encrypting anything.

Unbreakability of One Time Pads

The ciphertext on the previous slide is (literally) unbreakable. The reason is that the plaintext could be (literally) anything. For any proposed plaintext, there is a key that would produce the given ciphertext; and since the key is random, any key is just as likely as any other.

So it could be: comeroundatseven but it could equally be: dontbotherimbusy or anything else of the right length. We have nothing to go on (except length).

Stream Ciphers

Encrypt each byte/bit in turn – instead of in blocks. Loosely based around the idea of the one-time pad. XOR cipher where the key is as long as the message. Perfect and provable security. Generate a pseudorandom key stream and use it as the key to an XOR cipher. However, a one-time pad requires a truly random key stream (pad) to be provably secure. So stream ciphers can be completely insecure.

Synchronous Stream Ciphers

Pseudo-random digits generated independently of plaintext and ciphertext. Both sides must be synchronized.

Self-synchronising Stream Ciphers

Will resynchronize after a certain number of bytes since a lost one.

Basic Stream Ciphers

LFSR

Fairly easy to cryptanalyze (in basic form). With known plaintext and LFSR can calculate state and hence future stream. If unknown LFSR, then Berlekamp-Massey algorithm. Famously used in DVD CSS and A5/1. Wikimedia Commons

RC4

Designed in 1987 by Ron Rivest – kept secret. Exceptionally simple and compact stream cipher (pseudorandom number generator). Two stages: A key scheduling phase and a stream generation phase.

RC4 Pseudo-random Generation Algorithm (PRGA)


Aka key stream generator
i := 0
j := 0
while GeneratingOutput:
i := (i + 1) mod 256
j := (j + S[i]) mod 256
swap values of S[i] and S[j]
K := S[(S[i] + S[j]) mod 256]
output K
endwhile

Cryptanalysis of RC4

Remember – the same key can never be used twice as it’ll produce the same key stream. Andrew Roos experimentally showed that the first byte of the keystream is correlated to just the first three bytes of the key. The second output byte is biased to zero with a probability of 1/128. The first few bytes of the keystream are strongly non-random and related to the key.

Looking Back and Forward

Symmetric cryptography uses the same key (shared by two or more parties) used to encrypt and decrypt (e.g., AES). The hybrid approach utilizes the advantages of both symmetric and asymmetric cryptography. Asymmetric cryptography uses two different keys: a secret key (known to only one party) used to decrypt messages that were encrypted using the public key (known to all), e.g., RSA. It enables the creation of digital signatures.
